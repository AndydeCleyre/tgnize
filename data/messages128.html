In reply to this message
It's a tool to rewrite history. It takes range of commits, and recreates them, possibly on another branch. Commits don't have to be recreted as-is — you can reorder them, drop them, squash them, improve commit messages, etc.

It has multiple practical uses. I can think of 2 most common ones:

1. It's an alternative to merging via merge commit — instead of merging A into B, you can recreate commits from A on branch B.

Related, git pull --rebase. In case there is new commit(s) both in your local repo, and upstream, it'll rebase your work onto upstream instead of creating random-looking merge commit.

2. git rebase -i --autosquash works very well with git commit --fixup. They basically allow you to amend (improve) commit which is not the newest one. When you notice you screwed something in a commit which is not the latest, commit your improvements as a fixup to that commit. Then, git rebase -i --autosquash will automatically pick it up and will squash those 2 commits.
